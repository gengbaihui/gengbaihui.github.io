### 一、Dubbo简介
Dubbo是最有名的RPC服务调用框架，他是阿里开源的一个SOA服务治理框架，功能较为完善，支持多种传输和序列化方案。Dubbo最常用的应用就是远程调用。


SOA（Service-Oriented Architecture，面向服务架构）是一种强调服务重用和集成的架构模式。其核心思想是将应用拆分为多个松耦合的服务（Service），每个服务提供特定的业务能力，并通过标准化协议（如 SOAP、REST、消息队列等）进行通信。

官网：https://dubbo.apache.org/en/index.html


### 二、什么是RPC

RPC（Remote Procedure Call，远程过程调用）是一种计算机通信技术与编程范式，核心是让程序能像调用本地函数 / 方法一样，调用远程地址空间（如另一台服务器）的函数 / 方法，无需关注底层网络通信细节。

#### 2.1 RPC核心要点
- 透明性：屏蔽网络传输（如 TCP/UDP）、数据序列化 / 反序列化、服务寻址等底层细节，调用方感知不到 “远程” 特性。
- 客户机 - 服务器模式：请求方为客户端，提供服务方为服务器，通过请求 - 响应模式完成交互。
- 跨进程 / 跨机器：适用于分布式系统，解决不同进程、不同主机间的通信问题，无需共享内存。
- 规范与实现分离：RPC 是一套通信规范，有多种框架实现，如 Dubbo、gRPC、Thrift 等。

#### 2.2 细节

    - 序列化 / 反序列化	将对象转为字节流传输，接收端还原，常用协议有 Protobuf、JSON、Hessian 等
    - 服务发现	客户端获取服务端地址（如 Nacos、Eureka），支持集群部署与负载均衡
    - 通信协议	底层传输协议，如 TCP（可靠）、UDP（高效），部分框架自定义协议（如 Dubbo 协议）
    - 负载均衡	客户端选择最优服务端节点，提升系统可用性与性能
    - 容错机制	处理网络超时、服务不可用等问题，如超时重试、熔断、降级
  
  #### 2.3 技术选型
    - RPC 分布式系统的内部调用 HTTP 对外API    
    - 不同框架支持的语言和用的协议也都不同，序列化的方式也不同
    - 性能上 RPC框架用二进制序列化的方式可以传输更高效
  #### 2.4 不同框架序列化的方式
    - Protobuf
    - JSON
    - Thrift 二进制
    - Hessian2

┌──────────────────────── 业务层 ───────────────────────┐
│  本地方法调用  │  RPC远程调用（Dubbo/gRPC/Thrift）   │
├─────────────────────────────────────────────────────┤
│  RPC框架封装层 （含序列化、服务发现、负载均衡等）    │
├─────────────────── 应用层协议 ───────────────────────┤
│  自定义RPC协议（Dubbo协议） │ HTTP/2 │ HTTP/1.1     │
├─────────────────── 传输层协议 ───────────────────────┤
│                        TCP                          │
├─────────────────── 网络层/数据链路层/物理层 ──────────┤
│                        IP/以太网/光纤等              │

#### 2.5 核心层级逻辑说明
- RPC 是「应用层的调用范式」：落地在业务层 + 专属封装层，不属于 TCP/HTTP 这类标准化的网络协议层级，只是基于下层协议做封装实现。
- 底层必须依赖标准化网络协议：所有 RPC 框架最终都要基于TCP（绝大多数）做数据传输，部分框架（如 gRPC）会在 TCP 之上再封装一层HTTP/2，Dubbo 则直接在 TCP 上封装自定义的Dubbo 协议。
- 框架是 RPC 范式的具体实现：Dubbo、gRPC、Thrift 不是协议，是实现了 RPC 范式的框架，它们会选择 / 自定义下层协议，同时封装序列化、服务治理等 RPC 必备能力。
- 简单记：RPC 是 “调用逻辑”，协议是 “传输规则”，框架是 “把逻辑落地的工具”，工具必须基于规则才能跑通。
